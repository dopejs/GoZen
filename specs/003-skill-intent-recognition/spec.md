# Feature Specification: Skill-Based Intent Recognition

**Feature Branch**: `003-skill-intent-recognition`
**Created**: 2026-02-28
**Status**: Draft
**Input**: User description: "我们需要好好看看bot了，bot的意图识别一直没有做得很好，我希望能够看看是否有可能用skill来改善意图识别的准确率"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Skill 定义与注册 (Priority: P1)

作为 GoZen 管理员，我希望能够定义和注册 Skill（技能），每个 Skill 包含触发条件描述和对应的处理动作，这样 Bot 在收到用户消息时可以基于 Skill 进行更精准的意图匹配，而不是仅依赖硬编码的正则表达式。

**Why this priority**: 这是整个功能的基础。没有 Skill 定义机制，后续的意图匹配改进无从谈起。当前 Bot 的 NLU 仅支持固定的正则模式匹配，无法覆盖自然语言的多样性表达（如"帮我暂停一下"无法匹配 `pause`），导致大量合理请求被错误归类为闲聊。

**Independent Test**: 通过配置文件或管理界面创建一个 Skill，验证系统能正确加载并识别该 Skill 的存在。

**Acceptance Scenarios**:

1. **Given** 管理员定义了一个名为"进程控制"的 Skill，包含触发描述和对应动作, **When** 系统启动或重新加载配置, **Then** 该 Skill 被成功注册并可用于意图匹配
2. **Given** 管理员定义了多个 Skill, **When** 查看已注册的 Skill 列表, **Then** 所有 Skill 按优先级排序显示，包含名称、描述和状态
3. **Given** 管理员定义了一个无效的 Skill（缺少必要字段）, **When** 系统尝试加载, **Then** 系统报告具体的验证错误并跳过该 Skill，不影响其他 Skill 的加载

---

### User Story 2 - Skill 驱动的意图识别 (Priority: P1)

作为 Bot 用户，我希望当我用自然语言发送指令时（如"帮我把那个任务停掉"、"看看现在有哪些进程在跑"），Bot 能够准确理解我的意图并执行正确的操作，而不是把所有非精确匹配的消息都当作闲聊处理。

**Why this priority**: 这是用户直接感知到的核心改进。当前系统将所有不匹配正则的消息归为 `IntentChat`，导致用户必须记住精确的命令格式。Skill 驱动的意图识别应该能理解同一意图的多种自然语言表达方式。

**Independent Test**: 向 Bot 发送一条自然语言指令（如"帮我暂停一下"），验证 Bot 正确识别为控制意图并执行暂停操作，而非当作闲聊回复。

**Acceptance Scenarios**:

1. **Given** 已注册"进程控制"Skill, **When** 用户发送"帮我把任务停掉", **Then** Bot 识别为控制意图（stop），而非闲聊
2. **Given** 已注册多个 Skill, **When** 用户发送一条消息, **Then** 系统按 Skill 优先级依次匹配，返回置信度最高且超过阈值的结果
3. **Given** 用户发送的消息不匹配任何 Skill, **When** 所有 Skill 匹配置信度均低于阈值, **Then** 系统回退到现有行为（归类为闲聊意图）
4. **Given** 已注册 Skill, **When** 用户发送精确命令（如"pause"）, **Then** 现有的正则匹配优先生效，Skill 匹配作为补充而非替代

---

### User Story 3 - Skill 管理与调试 (Priority: P2)

作为 GoZen 管理员，我希望能够通过 Web UI 管理 Skill（查看、创建、编辑、启用/禁用、删除、测试），并能看到意图匹配的调试信息，以便持续优化 Skill 的识别准确率。

**Why this priority**: 管理和调试能力是持续改进意图识别的关键。没有可观测性，管理员无法知道哪些消息被错误分类，也无法针对性地优化 Skill 配置。

**Independent Test**: 通过 Web UI 查看 Skill 列表，对某条测试消息运行意图匹配，查看每个 Skill 的匹配得分和最终结果。

**Acceptance Scenarios**:

1. **Given** 管理员打开 Skill 管理界面, **When** 查看 Skill 列表, **Then** 显示所有已注册 Skill 的名称、状态、匹配统计（命中次数、准确率）
2. **Given** 管理员输入一条测试消息, **When** 执行意图匹配测试, **Then** 显示每个 Skill 的匹配得分、最终选中的意图、以及匹配过程的详细日志
3. **Given** 管理员禁用某个 Skill, **When** 后续消息到达, **Then** 该 Skill 不参与意图匹配，其他 Skill 正常工作

---

### User Story 4 - 多语言意图识别 (Priority: P3)

作为使用中文或其他非英语语言的 Bot 用户，我希望 Bot 能理解我用母语发出的指令，而不是只能识别英文命令。

**Why this priority**: 当前正则模式仅部分支持中文（如"批准"、"拒绝"、"人设"），大部分命令仅支持英文。Skill 系统应天然支持多语言意图描述，降低非英语用户的使用门槛。

**Independent Test**: 用中文发送"查看所有进程状态"，验证 Bot 正确识别为查询状态意图。

**Acceptance Scenarios**:

1. **Given** Skill 定义中包含中文触发描述, **When** 用户用中文发送匹配消息, **Then** Bot 正确识别意图
2. **Given** 同一 Skill 包含多语言触发描述, **When** 用户用任一支持语言发送消息, **Then** Bot 均能正确识别

---

### Edge Cases

- 当用户消息同时匹配多个 Skill 且置信度相近时，系统如何决策？（应选择优先级最高的 Skill）
- 当 Skill 匹配服务不可用或超时时，系统如何处理？（应回退到现有正则匹配，不阻塞消息处理）
- 当用户消息极短（如单个字符或表情符号）时，Skill 匹配是否应该跳过？（应直接归类为闲聊，避免无意义的匹配开销）
- 当 Skill 配置在运行时被修改时，是否需要重启？（应支持热重载，无需重启）
- 当同一消息既匹配正则命令又匹配 Skill 时，优先级如何？（正则精确匹配优先，Skill 作为补充）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须支持定义 Skill，每个 Skill 包含：名称、描述、优先级、关联的意图类型、多语言关键词（触发条件）、启用/禁用状态
- **FR-002**: 系统必须在收到用户消息时，先执行现有正则模式匹配；若无精确匹配，再依次尝试已注册的 Skill 进行意图识别
- **FR-003**: 每次 Skill 匹配必须返回置信度分数，系统仅在置信度超过可配置阈值时才采纳匹配结果
- **FR-004**: 系统必须支持 Skill 的热重载——配置变更后无需重启即可生效
- **FR-005**: 系统必须在 Skill 匹配服务不可用时自动回退到现有正则匹配，确保消息处理不中断
- **FR-006**: 系统必须记录每次意图匹配的过程（输入消息、各 Skill 得分、最终结果），用于调试和优化
- **FR-007**: 系统必须支持通过 Web UI 查看、创建、编辑、启用/禁用、删除 Skill，并提供意图匹配测试功能和匹配日志查看
- **FR-008**: 系统必须支持 Skill 定义中包含多语言触发描述，使同一 Skill 能响应不同语言的用户输入
- **FR-009**: 系统必须提供一组内置 Skill，覆盖现有正则模式对应的意图（控制、绑定、审批、任务发送、人设、遗忘、状态查询），作为开箱即用的基础能力
- **FR-010**: Skill 意图匹配采用混合模式——本地快速匹配（关键词+同义词+模糊匹配）优先处理，当本地匹配置信度低于阈值时回退到 LLM 语义理解。大部分请求由本地匹配完成（延迟可忽略），仅模糊情况调用 LLM（延迟不超过 2 秒），整体成本可控
- **FR-011**: Skill 匹配仅负责识别意图类型（如"控制"、"绑定"、"查询状态"），具体的动作参数（action、target 等）由后续 LLM 处理流程从原始消息中提取

### Key Entities

- **Skill**: 意图识别的基本单元。包含名称、描述、优先级、关联意图类型、触发条件、启用状态、匹配统计
- **MatchResult**: 单次意图匹配的结果。包含匹配的 Skill、置信度分数、解析出的意图参数
- **MatchLog**: 意图匹配的调试记录。包含输入消息、各 Skill 的匹配得分、最终选中结果、耗时

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 自然语言指令的正确识别率从当前水平提升至 85% 以上（以预定义的测试用例集衡量）
- **SC-002**: 用户无需记忆精确命令格式，使用自然语言表达即可完成 90% 的常用操作
- **SC-003**: 现有精确命令（如 `pause`、`bind`）的识别准确率保持 100%，不受 Skill 系统影响
- **SC-004**: 95% 的意图匹配请求（本地匹配）额外延迟不超过 500 毫秒；LLM 回退场景延迟不超过 2 秒
- **SC-005**: 管理员能在 5 分钟内完成一个新 Skill 的创建和测试
- **SC-006**: 中文和英文指令的意图识别准确率差异不超过 5 个百分点

## Clarifications

### Session 2026-02-28

- Q: Skill 匹配后如何提取动作参数（action、target 等）？ → A: Skill 仅负责识别意图类型，参数提取交给后续 LLM 处理（利用已有的 Bot LLM 客户端）
- Q: LLM 回退的成本控制策略？ → A: 复用现有 guardrails 预算控制（spending cap、rate limit），不设独立限制
- Q: SC-004 与 FR-010 延迟指标冲突如何解决？ → A: 分层目标——本地匹配 ≤500ms，LLM 回退 ≤2s；SC-004 改为"95% 请求 ≤500ms，LLM 回退 ≤2s"

## Assumptions

- 当前正则匹配逻辑将保留作为第一优先级的快速路径，Skill 系统作为补充层
- 内置 Skill 将覆盖现有所有意图类型，确保向后兼容
- Skill 配置存储在现有的 `zen.json` 配置文件中，遵循现有的配置管理模式
- 意图匹配的置信度阈值默认为 0.7，可由管理员调整
- Skill 相关的 LLM 调用（意图回退匹配 + 参数提取）复用现有 Bot LLM 客户端和 profile，受已有 guardrails 预算控制约束，不设独立限制
