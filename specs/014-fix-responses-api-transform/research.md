# Research: OpenAI Responses API Format

**Feature**: 014-fix-responses-api-transform
**Date**: 2026-03-07

## Decision 1: Error Detection Strategy

**Decision**: Use substring match on `"input is required"` in response body text, regardless of HTTP status code (observed as 500 from production).

**Rationale**: The production error from `code.b886.top` returns `{"error":{"message":"input is required ...","type":"new_api_error"}}` with HTTP 500. The error type varies by provider (`new_api_error` vs `invalid_request_error`), but the message `"input is required"` is consistent as it's generated by the upstream framework when the `input` field is missing from the Responses API request body.

**Alternatives considered**:
- Exact match on error type `"new_api_error"` â€” rejected: too brittle, provider-specific
- Check HTTP status code (only 400) â€” rejected: production shows 500
- Regex pattern â€” rejected: unnecessary complexity for a simple substring match

## Decision 2: Retry Scope

**Decision**: Retry is internal to a single provider attempt. It does NOT consume a failover slot or mark the provider unhealthy.

**Rationale**: The "input is required" error is not a provider health issue â€” the provider is working correctly, just expecting a different format. Marking it unhealthy or failing over would be wrong behavior. The retry is a format negotiation within the same provider.

**Alternatives considered**:
- Separate provider attempt â€” rejected: wastes failover budget, confusing semantics
- Pre-detect format via config option â€” rejected: scope creep, user must manually configure
- Cache provider format after first success â€” rejected: adds state management complexity, deferred to future optimization

## Decision 3: Body Transformation Strategy

**Decision**: Two-step transformation. `forwardRequest` already produced Chat Completions body from Anthropic. The retry transforms Chat Completions â†’ Responses API (a simpler transform since the formats are related).

**Rationale**: Reusing the existing Anthropicâ†’Chat Completions transform avoids duplicating complex logic (system message extraction, tool format conversion, etc.). The Chat Completionsâ†’Responses API transform is mostly field renames and tool flattening.

**Alternatives considered**:
- Direct Anthropicâ†’Responses API transform â€” rejected: duplicates shared logic with Chat Completions transform
- Add a `format` parameter to `forwardRequest` â€” rejected: complicates existing interface for a niche case

## Decision 4: Streaming Response Transform

**Decision**: Add a new `transformResponsesAPIToAnthropic` function in `stream.go` that handles Responses API SSE events (which use `event:` + `data:` lines with typed event names) and converts them to Anthropic SSE format.

**Rationale**: Responses API streaming uses a fundamentally different SSE format from Chat Completions. Chat Completions uses `data:` lines only with `[DONE]` sentinel. Responses API uses semantic `event:` lines (`response.output_text.delta`, `response.completed`, etc.) with JSON `data:` payloads. These require separate parsing logic.

**Alternatives considered**:
- Extend existing `transformOpenAIToAnthropic` to handle both formats â€” rejected: would make the function too complex with format detection branching
- Convert Responses API stream to Chat Completions stream first, then reuse existing transform â€” rejected: unnecessary intermediate step, adds latency

## OpenAI Responses API Key Facts

### Request Format (`POST /v1/responses`)
- `input`: String or array of message objects (replaces `messages`)
- `model`: Required, same as Chat Completions
- `stream`: Boolean, same semantics
- `max_output_tokens`: Replaces `max_completion_tokens`/`max_tokens`
- `tools`: Flattened format â€” `{type:"function", name:"...", parameters:{...}}` (no nested `function` wrapper)
- `instructions`: Top-level system prompt field
- `store`: Boolean (default true), controls server-side storage

### Response Format
- `output`: Array of typed items (replaces `choices`)
  - `type: "message"` â€” text content in `.content[].type:"output_text"`
  - `type: "function_call"` â€” tool call with `.name`, `.arguments`, `.call_id`
- `status`: `"completed"`, `"incomplete"`, `"failed"`
- `usage`: `{input_tokens, output_tokens, total_tokens}`

### Streaming Events
- Uses `event:` + `data:` lines (not just `data:` like Chat Completions)
- Key events: `response.created`, `response.output_text.delta`, `response.function_call_arguments.delta`, `response.completed`
- No `[DONE]` sentinel â€” completion signaled by `response.completed` event
- Events have `sequence_number` for ordering
